# Техническое задание для Codex

## Проект
Локальная система аналитики рынка на основе данных Bybit.

Система состоит из двух основных частей:
1. **Python-ядро** для загрузки, проверки и расчёта рыночных данных.
2. **MCP-сервер** поверх этого ядра для работы из ChatGPT.

Цель: пользователь общается с системой внутри ChatGPT, ChatGPT вызывает MCP tools, Python-ядро получает данные с Bybit, считает признаки и возвращает структурированный результат для интерпретации.

---

## 1. Общая цель

Нужно реализовать локальную систему, которая по запросу пользователя:
- получает рыночные данные с Bybit;
- валидирует данные;
- считает заранее определённый набор индикаторов и производных признаков;
- формирует компактный результат в JSON;
- отдаёт этот результат через MCP tools для использования внутри ChatGPT.

Ключевой принцип:
**Python считает, ChatGPT интерпретирует.**

LLM не должна:
- сама ходить в Bybit API;
- сама рассчитывать индикаторы;
- сама подменять отсутствующие данные;
- использовать mock/synthetic данные без явной отдельной команды тестового режима.

---

## 2. Цели MVP

На первом этапе система должна уметь:
- анализировать один или несколько символов Bybit;
- работать с OHLCV по свечам;
- поддерживать базовые таймфреймы;
- считать ограниченный набор индикаторов;
- возвращать компактный аналитический snapshot;
- предоставлять доступ к этим данным через MCP.

На этапе MVP **не требуется**:
- автоторговля;
- управление ордерами;
- сложная работа со стаканом;
- работа 24/7 в фоне;
- веб-интерфейс;
- сложная БД;
- переобучение моделей;
- генерация решений на стороне LLM.

---

## 3. Базовый пользовательский сценарий

Пользователь находится в ChatGPT и пишет, например:
- «Разбери BTCUSDT на 1H»
- «Сравни BTCUSDT и ETHUSDT на 15m»
- «Проверь, есть ли трендовый сценарий по SOLUSDT»

ChatGPT вызывает MCP tool.
MCP tool вызывает Python-ядро.
Python-ядро:
- загружает данные,
- валидирует,
- считает признаки,
- возвращает JSON.
Далее ChatGPT строит ответ на основании этого JSON.

## 6. Источник данных

### 6.1. Bybit API
Использовать Bybit API для получения свечей.

Минимально необходимо:
- загрузка OHLCV по символу;
- выбор таймфрейма;
- загрузка ограниченного окна свечей.

### 6.2. Ограничение по данным
Система не должна автоматически грузить «всю историю».

Нужно реализовать режим загрузки только нужного окна.
Например:
- для EMA200 загружать окно с запасом;
- для анализа брать ограниченное число свечей;
- значение window_size должно задаваться явно.

### 6.3. Поддерживаемые таймфреймы MVP
Минимум:
- 5m
- 15m
- 1h
- 4h

---

## 7. Валидация данных

После загрузки данных обязательно выполнять валидацию.

### 7.1. Проверки
Нужно проверить:
- данные не пустые;
- количество свечей не меньше минимально допустимого;
- timestamps отсортированы по возрастанию;
- нет дубликатов timestamps;
- нет пропусков в обязательных колонках;
- колонки OHLCV имеют корректные типы;
- high >= low;
- open и close находятся в допустимом диапазоне;
- volume не отрицательный;
- таймфрейм соответствует ожидаемому шагу между свечами.

### 7.2. Правило отказа
Если хотя бы одна критичная проверка не проходит:
- анализ останавливается;
- возвращается явная ошибка;
- никаких synthetic/mock данных не подставлять.

Ключевой принцип:
**нет валидных данных — нет аналитики.**

## 8. Индикаторы и признаки

### 8.1. Важно
Не считать все индикаторы библиотеки подряд.
Нужно реализовать **ограниченное ядро признаков**, а не бесконечный набор.

### 8.2. Обязательный набор индикаторов для MVP

Важно: текущий набор индикаторов нужно рассматривать как **стартовое ядро для MVP**, а не как финальную аналитическую модель.

Codex должен реализовать этот набор так, чтобы его можно было **безболезненно заменить, расширить или частично отключить** в будущем.

Требования к модулю индикаторов и признаков:
- модуль должен быть изолирован от остальной логики проекта;
- набор рассчитываемых индикаторов не должен быть жёстко размазан по коду;
- должна быть возможность централизованно менять состав индикаторов;
- должна быть возможность добавлять новые индикаторы без переписывания MCP layer и основного data pipeline;
- производные признаки должны строиться поверх явного и понятного слоя базовых метрик;
- текущая реализация должна восприниматься как **заменяемое тестовое ядро**, на основе которого потом можно внедрить более сложную пользовательскую логику.

Требование по реализации:
- ключевые индикаторы для MVP считать **вручную на pandas / numpy**, где это разумно и просто;
- не завязывать ядро на тяжёлую или критичную магию сторонней индикаторной библиотеки;
- сторонняя библиотека допустима как вспомогательная, но базовые расчёты должны оставаться прозрачными и легко проверяемыми;
- приоритет: прозрачность и контроль расчётов выше, чем количество готовых функций.

Желательно предусмотреть один из вариантов:
- конфигурируемый реестр индикаторов;
- отдельный IndicatorService с подключаемыми расчётными блоками;
- декларативное описание активных индикаторов и признаков.

На этапе MVP достаточно реализовать стартовый набор ниже.

#### Тренд
- EMA 20
- EMA 50
- EMA 200

#### Моментум
- RSI 14

#### Волатильность
- ATR 14

#### Объём
- relative volume

#### Контекст
- VWAP
- distance_to_vwap_pct

### 8.3. Правило расчёта VWAP для MVP

Для MVP использовать **сессионный / intraday VWAP** как основной вариант.

Это считать каноническим поведением.

Правила:
- rolling-VWAP не использовать как основную реализацию по умолчанию;
- если по техническим причинам понадобится временный упрощённый вариант, он должен быть явно назван как `rolling_vwap` и не маскироваться под классический VWAP;
- в snapshot должно быть понятно, какой именно вариант VWAP был использован;
- для 1h и 4h VWAP на этапе MVP использовать как контекстный признак, а не как единственный или решающий сигнал.

### 8.4. Производные признаки
На основе индикаторов нужно считать компактные признаки:
- price_above_ema20
- price_above_ema50
- price_above_ema200
- ema20_above_ema50
- ema50_above_ema200
- trend_direction
- trend_strength
- rsi_state
- atr_state
- volume_state
- price_vs_vwap_state
- setup_flags
- risk_flags

### 8.4. Формат производных состояний
Состояния должны быть нормализованы в ограниченный словарь значений.

Примеры:
- trend_direction: `bullish | bearish | neutral`
- trend_strength: `weak | moderate | strong`
- rsi_state: `oversold | neutral | strong | overbought`
- volume_state: `low | normal | elevated`
- price_vs_vwap_state: `below | near | above | extended_above | extended_below`

## 9. Snapshot Builder

Нужно собрать единый компактный JSON-результат для передачи в ChatGPT.

### 9.1. Пример структуры

```json
{
  "symbol": "BTCUSDT",
  "timeframe": "1h",
  "as_of": "2026-04-23T18:30:00Z",
  "price": 68420.5,
  "indicators": {
    "ema20": 68110.2,
    "ema50": 67580.4,
    "ema200": 64990.1,
    "rsi14": 62.4,
    "atr14": 740.3,
    "distance_to_vwap_pct": 0.8,
    "relative_volume": 1.34
  },
  "states": {
    "trend_direction": "bullish",
    "trend_strength": "moderate",
    "rsi_state": "strong",
    "atr_state": "normal",
    "volume_state": "elevated",
    "price_vs_vwap_state": "above"
  },
  "flags": {
    "setup_flags": ["trend_continuation", "pullback_candidate"],
    "risk_flags": ["near_local_resistance"]
  },
  "meta": {
    "bars_used": 300,
    "data_source": "bybit",
    "validation_passed": true
  }
}
```

### 9.2. Требования к snapshot
- структура должна быть стабильной;
- поля должны быть типизированы;
- отсутствие значения должно обрабатываться явно;
- snapshot должен быть удобен для чтения LLM и человеком.

### 9.3. Канонический формат ответа MCP

Для MVP использовать **единую обёртку ответа**.

Успешный ответ:

```json
{
  "status": "ok",
  "data": { ... snapshot ... }
}
```

Ответ с ошибкой:

```json
{
  "status": "error",
  "error": {
    "code": "...",
    "message": "...",
    "details": { ... }
  }
}
```

Правила:
- не смешивать несколько форматов успешного ответа;
- не возвращать «голый snapshot» в одном tool и обёртку в другом;
- все MCP tools должны использовать единый контракт ответа.

## 11. Обработка ошибок

### 11.1. Критические ошибки
Критическими считать:
- API недоступно;
- пустой ответ;
- недостаточно свечей;
- битые данные;
- ошибка расчёта индикаторов;
- ошибка сериализации snapshot.

### 11.2. Правила обработки
При критической ошибке:
- не использовать fallback на synthetic данные;
- не возвращать фальшивую аналитику;
- возвращать структурированную ошибку;
- ошибка должна быть понятна для ChatGPT и пользователя.

Пример:

```json
{
  "status": "error",
  "error_code": "INSUFFICIENT_DATA",
  "message": "Недостаточно свечей для расчёта EMA200",
  "details": {
    "symbol": "BTCUSDT",
    "timeframe": "1h",
    "required_bars": 220,
    "received_bars": 140
  }
}
```

## 12. Требования к MCP-серверу

### 12.0. Подключение ChatGPT к локальному MCP на этапе MVP

На этапе MVP MCP-сервер предполагается запускать **локально на машине пользователя**.

Для подключения ChatGPT к локальному MCP-серверу нужно использовать **Cloudflare Tunnel**.

Это нужно зафиксировать как базовый способ локальной разработки и первичного подключения.

Требования:
- локальный MCP-сервер должен подниматься на localhost;
- наружу он должен публиковаться через Cloudflare Tunnel;
- в README нужно описать минимальный сценарий запуска через `cloudflared`;
- решение через Cloudflare Tunnel считать основным для MVP;
- использование ngrok не является базовым вариантом и не должно быть основным сценарием в документации.

Важно:
- на этапе MVP не требуется сразу выносить MCP на VPS или постоянный удалённый сервер;
- архитектура при этом должна позволять в будущем без серьёзной переделки перенести MCP на удалённый сервер.

## 12. Требования к MCP-серверу

Нужно реализовать MCP-сервер поверх Python-ядра.

Сервер должен предоставлять набор инструментов для ChatGPT.

### 12.1. Транспорт MCP для MVP

Целевым транспортом для MVP считать **HTTP-совместимый вариант (Streamable HTTP / HTTP transport)**.

Требования:
- не ориентироваться на SSE как на основной транспорт;
- реализация должна быть совместима с запуском через Cloudflare Tunnel;
- транспорт должен быть простым и предсказуемым для локальной разработки и дальнейшего выноса на удалённый сервер.

Итог:
- для MVP брать **HTTP-совместимый MCP transport** как основной;
- SSE не использовать как базовый вариант, если он не требуется библиотекой или фреймворком.

### 12.2. Основные MCP tools для MVP

#### 1. analyze_symbol
Выполняет анализ одного символа на одном таймфрейме.

Вход:
- symbol
- timeframe
- bars_limit (опционально)

Выход:
- компактный snapshot JSON

#### 2. compare_symbols
Сравнивает несколько символов на одном таймфрейме.

Вход:
- symbols[]
- timeframe
- bars_limit (опционально)

Выход:
- массив snapshot-объектов
- краткие агрегированные поля для сравнения

#### 3. scan_watchlist
Проверяет список символов и возвращает базовый shortlist.

Вход:
- symbols[]
- timeframe
- scan_mode

Выход:
- список результатов
- priority / score / flags

Для MVP использовать **простой, прозрачный и rule-based скоринг**, а не сложную эвристику.

Скоринг должен строиться на ограниченном наборе признаков:
- направление тренда;
- сила тренда;
- RSI state;
- relative volume;
- положение цены относительно VWAP;
- наличие признаков растянутости цены;
- наличие базовых setup flags.

Требования к scan_watchlist:
- score должен быть объяснимым и воспроизводимым;
- нельзя использовать «чёрный ящик» без явной логики;
- вклад каждого фактора должен быть описан в коде и README;
- результат должен включать не только score, но и flags, объясняющие причину оценки.

Минимальный формат:
- score: числовая оценка, например 0–100;
- priority: `high | medium | low`;
- flags: список кратких причин, например `trend_ok`, `elevated_volume`, `overextended`, `weak_trend`, `near_vwap`.

#### 4. get_raw_snapshot
Возвращает технический snapshot без дополнительной агрегации.

Вход:
- symbol
- timeframe

Выход:
- полный JSON snapshot

## 21. Критерии приёмки

Система считается принятой, если:

1. По запросу можно получить валидный snapshot по символу и таймфрейму.
2. При невалидных данных возвращается ошибка, а не выдуманный результат.
3. Индикаторы считаются локально в Python.
4. MCP tools вызываются из ChatGPT без ручных правок кода между запросами.
5. Ответы tool'ов имеют стабильную JSON-структуру.
6. Есть тесты на критические сценарии.
7. Есть README с инструкцией запуска.
8. В проекте отсутствует смешивание боевого режима и synthetic test data.

Нужно отдельно проследить за следующим:

## 24. Инструкция для Codex по стилю работы

При реализации проекта Codex должен:
- двигаться по этапам;
- после каждого крупного этапа кратко фиксировать, что сделано;
- не придумывать mock-данные для боевого режима;
- не подменять ошибки «правдоподобными результатами»;

- не раздувать объём индикаторов сверх указанного ядра;
- при необходимости предлагать улучшения отдельно, не смешивая их с MVP.
